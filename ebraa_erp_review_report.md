### 4. تعارض عزل السيريال بين Tenants

| الملف | ما يقوله |
|---|---|
| `05_الاختبارات` (INV-03) | "نفس السيريال لـ Tenant مختلف → مسموح (السيريال فريد per Tenant)" |
| `06_معمارية_قاعدة_البيانات` (Schema) | `UNIQUE (serial_number)` بدون `tenant_id` في جدول `device_items` |

**الأثر:** لو الـ Unique Constraint على `serial_number` فقط، إذن لا يمكن لمحلين مختلفين إضافة نفس السيريال — وده متعارض مع قاعدة الاختبار INV-03. في الواقع، نفس الموبايل ممكن يُباع ويُشترى في محلين مختلفين.

**التوصية:** غيّر الـ Constraint في Schema لـ `UNIQUE (serial_number, tenant_id)` وليس `UNIQUE (serial_number)` فقط. لكن بما إن في Dual DB (كل tenant له DB منفصلة)، الـ UNIQUE على `serial_number` وحدها كافية داخل كل DB — وده يخلي INV-03 خاطئ. **يجب توضيح هذه النقطة في التوثيق.**

## 🟡 ثانياً: النقاط الناقصة والفجوات الوظيفية

### 1. Schema ناقص لموديول الصيانة
لا يوجد في ملف قاعدة البيانات (`06`) أي جداول للصيانة، بينما صفحة الخدمة موثقة بالكامل.

**الجداول المفقودة المطلوبة:**
```sql
CREATE TABLE repair_tickets (
    id, branch_id, customer_id,
    device_brand, device_model, serial_number,
    problem_description TEXT,
    physical_condition TEXT,        -- خدوش، كسور
    estimated_cost DECIMAL(15,2),
    actual_cost DECIMAL(15,2),
    status VARCHAR(20),             -- received, diagnosing, repairing, ready, delivered
    received_at TIMESTAMP,
    estimated_ready_at TIMESTAMP,
    delivered_at TIMESTAMP,
    technician_id BIGINT REFERENCES users(id),
    created_at TIMESTAMP
);

CREATE TABLE repair_parts (         -- قطع الغيار المستخدمة
    id, ticket_id, product_id,
    quantity DECIMAL(15,3),
    unit_cost DECIMAL(15,2)
);
```

---

### 2. جدول المصروفات (Expenses) مفقود من Schema
صفحة الإدارة المالية (`10`) تذكر "إدارة المصروفات التشغيلية (إيجار، كهرباء، مرتبات)" لكن لا يوجد جدول `expenses` في ملف قاعدة البيانات.

**الجدول المفقود:**
```sql
CREATE TABLE expenses (
    id              BIGSERIAL PRIMARY KEY,
    branch_id       BIGINT REFERENCES branches(id),
    treasury_id     BIGINT REFERENCES treasuries(id),
    category        VARCHAR(100),   -- إيجار، كهرباء، مرتبات، صيانة
    amount          DECIMAL(15,2) NOT NULL,
    description     TEXT,
    date            DATE NOT NULL,
    created_by      BIGINT REFERENCES users(id),
    created_at      TIMESTAMP DEFAULT NOW()
);
```

---

### 3. جدول الأقساط ناقص تفاصيل ربطه بطريقة الدفع
جدول `installments` موجود في Schema لكن في جدول `sales` مفيش حقل `payment_type` واضح يبيّن إن الفاتورة دي آجل أصلاً. كمان مفيش `payment_transactions` table لتسجيل دفعات متعددة على نفس الفاتورة.

**الجداول/الحقول المقترحة:**
```sql
-- في جدول sales: إضافة
payment_type VARCHAR(20), -- cash, credit, partial, network
paid_amount  DECIMAL(15,2) DEFAULT 0,
remaining    DECIMAL(15,2) DEFAULT 0,

-- جدول جديد للدفعات
CREATE TABLE payment_transactions (
    id, sale_id,
    amount DECIMAL(15,2),
    payment_method VARCHAR(20), -- cash, network, installment
    reference_number VARCHAR(100),
    paid_at TIMESTAMP
);
```

### 5. نظام الإشعارات غير مكتمل التوثيق
مذكور في أكثر من مكان (Telegram Bot، Laravel Reverb) لكن مفيش توثيق واضح لـ:
- هيكل جدول `notifications` (لو هيخزن الإشعارات)
- الفرق بين الـ Real-time (Reverb) والـ Async (Telegram)
- إشعارات الأقساط المتأخرة: كيف تُجدوَل؟ (Scheduled Job؟)
- لو العميل مش بيستخدم تليجرام — في بديل؟ (SMS؟)

### 6. Onboarding Flow ناقص في التوثيق
`STP-041` في الباكلوج بيذكر "خطوات إعداد المحل الجديد (Wizard)" لكن مفيش ملف توثيق لهذه الشاشة ومفيش صفحة في الـ pages folder.

**الخطوات المقترحة للـ Wizard:**
1. بيانات الشركة (الاسم، الشعار، الضريبة)
2. إنشاء أول فرع ومخزن
3. إضافة أول منتج (أو استيراد Excel)
4. إضافة أول موظف وتحديد صلاحياته
5. اختبار أول عملية بيع

### 7. حالة `in_repair` للسيريال مذكورة في الاختبارات لكن غير محددة في Schema
في `05_الاختبارات` (POS-02): "بيع سيريال حالته `in_repair` → رسالة خطأ"  
لكن في Schema جدول `device_items`، القيم المذكورة هي: `available, sold, in_transit`  
**`in_repair` غير موجود** في enum الحالات.

**التصحيح المطلوب:**
```sql
status VARCHAR(20) DEFAULT 'available'
-- القيم: available, sold, in_transit, in_repair, reserved
```

---

### 8. ربط تذكرة الصيانة بالمخزن غير موثق تقنياً
صفحة الصيانة تقول "يتم سحب قطع الغيار آلياً من مخزن الفرع" لكن آلية الربط غير موثقة:
- هل يتم سحب قطع الغيار عند فتح التذكرة؟ أم عند إغلاقها؟
- لو لم تُستخدم قطعة — يرجع للمخزن تلقائياً؟
- ماذا يحدث لو قطعة الغيار غير متوفرة في المخزن؟

---

### 9. صفحة الهبوط (Landing Page) موثقة في الفهرس لكن محتواها فارغ
ملف `01_صفحة_الهبوط_landing.md` مذكور في الفهرس لكن لم يتم تحميل محتواه — يُرجى مراجعة إذا كان يحتوي على مواصفات.

---

### 10. نظام الضمان غير موثق كـ Feature مستقلة
مذكور في عدة أماكن (US-06 ، Schema: `warranty_months`) لكن مفيش:
- منطق واضح لحساب تاريخ انتهاء الضمان (من تاريخ البيع؟ من تاريخ الصنع؟)
- صفحة أو section لإدارة مطالبات الضمان
- تقرير للأجهزة التي ستنتهي ضمانها قريباً
