## 2. منطق عزل البيانات المُصحَّح (Multi-tenancy)

> ⚠️ **تم تصحيح هذا القسم** — نعتمد الآن على حزمة **stancl/tenancy** لإدارة عزل البيانات تلقائياً.

```php
// استراتيجية stancl/tenancy (راجع الملف 11 للتفاصيل)

// الحزمة تتولى عملية التبديل بين الـ Database Connections تلقائياً بناءً على الـ Domain.
// يتم استخدام الـ TenantAware Middleware لضمان عزل البيانات.

abstract class TenantModel extends Model {
    use \Stancl\Tenancy\Database\Concerns\BelongsToTenant; // الربط التلقائي بالـ Tenant
}
```

**القاعدة الجديدة الذهبية:**
- `Tenant`, `Plan`, `Affiliate`, `SaasAdmin` → يرثون من `CentralModel`
- كل باقي الـ Models → يرثون من `TenantModel`

## 3. أنواع الأصناف وكيفية معالجتها

### أ. صنف بسيريال (Serialized Product)
- **مثال:** موبايل، رسيفر، لابتوب، أجهزة بلايستيشن، شاشات.
- **المنطق:** لكل قطعة مادية سجل خاص في `device_items` بسيريالها الفريد.

### ب. صنف بالكمية / سيريال عام (Bulk / General Serial)
- **مثال:** شواحن، قطع غيار صغيرة، إكسسوارات، أدوات مكتبية.
- **المنطق:** يُتابع بالكمية في جدول `products` أو بسيريال موحد للنوع.

### ج. منطق حساب التكلفة (Costing Logic) 🌟
يعتمد النظام على **التكلفة المتوسطة المتحركة (Moving Average Cost)** لضمان أدق حساب للأرباح:
- **المعادلة:** `(رصيد المخزن القديم × التكلفة القديمة + الكمية الجديدة × سعر الشراء الجديد) ÷ (الرصيد القديم + الكمية الجديدة)`.
- **التوقيت:** يتم تحديث التكلفة في جدول `products` مع كل عملية (شراء، مرتجع بيع، تسوية موجبة).
- **الثبات:** يتم تسجيل التكلفة في جدول `invoice_lines` وقت البيع لضمان عدم تغير أرباح الفواتير القديمة تاريخياً.

## 4. تدفق عملية البيع الكاملة (POS Workflow)

```
[1] البائع يفتح شاشة POS
        ↓
[2] يمسح الباركود أو يكتب السيريال / اسم المنتج
        ↓
[3] النظام يبحث في device_items (status = available)
    → إذا موجود: يضيفه للسلة مع سعر البيع
    → إذا Sold: يعطي خطأ "هذه القطعة تم بيعها بالفعل"
    → إذا غير موجود: يقترح إضافته للمخزن
        ↓
[4] تجميع السلة (يمكن خلط Serialized + Bulk)
        ↓
[5] تطبيق الخصم (بشرط: لا يتجاوز 100%)
        ↓
[6] اختيار طريقة الدفع:
    (نقدي) → تسجيل في cash_transactions
    (شبكة) → تسجيل مع رقم المرجع
    (آجل)  → إضافة للـ customer.total_debt
    (جزئي) → تسجيل جزء نقدي + جزء آجل
        ↓
[7] إتمام البيع (Transaction آمن):
    - إنشاء سجل في sales
    - إنشاء سجلات في sale_items
    - تحديث device_items → status = sold
    - تحديث stock_quantity للـ Bulk items
    - إصدار رقم فاتورة تسلسلي
        ↓
[8] طباعة الفاتورة (Thermal/A4)
```

## 5. منطق الجرد بالموبايل (Inventory Scanning)

```
[1] صاحب المحل يفتح صفحة "بدء الجرد" ويحدد المخزن
        ↓
[2] النظام يولد "جلسة جرد" (inventory_session) جديدة
        ↓
[3] الموظف يفتح الصفحة من الموبايل → تفعيل الكاميرا
        ↓
[4] عند مسح سيريال:
    → استخدام Inertia v3 `useHttp` لإرسال طلب خلفي {serial: "XXXXX"}
    → النظام يتحقق:
      [✓ موجود في device_items = available] → صوت نجاح، يُسجل
      [✗ sold/in_repair]                   → صوت تحذير، يُعلم
      [✗ غير موجود أصلاً]                 → صوت خطأ، يُضيف للـ "غير مسجل"
        ↓
[5] في النهاية: تقرير الفروقات والمطابقة.

## 5.5 منطق التنبيهات وإرسال الفواتير (Telegram Bot API)
بدلاً من الوسائل التقليدية المكلفة، يعتمد النظام على **تليجرام** للتواصل:
1. **ربط البوت:** يقوم صاحب المحل بربط Telegram Bot الخاص بالنظام.
2. **إرسال الفواتير:** عند إتمام البيع، يُرسل رابط الفاتورة (أو ملف PDF) آلياً لرقم العميل (عبر تليجرام).
3. **تنبيهات الإدارة:** إشعارات فورية لصاحب العمل عند:
   - تحقيق عملية بيع جديدة.
   - وصول منتج لحد الطلب (Low Stock).
   - محاولة دخول غير مصرح بها.```

## 7. قواعد العمل الصارمة (Business Rules)

| # | القاعدة | السبب |
|---|---|---|
| BR-01 | لا يمكن بيع سيريال حالته `sold` بدون مرتجع مسبق. | منع الغش والتلاعب. |
| BR-02 | لا يمكن حذف منتج له تاريخ مبيعات (Soft Delete فقط). | الحفاظ على سلامة التقارير التاريخية. |
| BR-03 | رقم الفاتورة تسلسلي لكل Tenant ولا يُعاد. | الامتثال المحاسبي. |
| BR-04 | البائع لا يمكنه تعديل سعر البيع المبرمج مسبقاً. | ضبط هامش الربح. |
| BR-05 | الخصم لا يتجاوز النسبة المسموح بها حسب دور المستخدم. | حماية هامش الربح. |
| BR-06 | لا يمكن حذف Tenant له عملاء أو مبيعات (Archive فقط). | حماية البيانات. |
| BR-07 | كل المبالغ تُخزّن بـ decimal(15,2) في قاعدة البيانات. | تجنب أخطاء الكسور العشرية (Floating Point). |

## 8. خريطة صلاحيات الأدوار المتقدمة (Advanced RBAC)

النظام يعتمد على **صلاحيات متناهية الدقة (Micro-Permissions)** مقسمة إلى مصفوفة عمليات (CRUD) لكل شاشة، بالإضافة إلى أدوار النظام المنفصلة:

### أ. مصفوفة العمليات لمستخدمي المحل (Tenant CRUD Matrix)
لكل موديول (فواتير، أصناف، حسابات، خزينة)، يتحكم النظام في:
- **استعراض (Read):** القدرة على فتح الشاشة ورؤية البيانات.
- **إضافة (Create):** إنشاء مستندات أو سجلات جديدة.
- **تعديل (Update):** تعديل البيانات الموجودة (مع قيود BR-02).
- **حذف (Delete):** الحذف (غالباً Soft Delete).

### ب. أعلام الميزات الخاصة (Feature Flags)
صلاحيات تشغيلية إضافية تُمنح للمحترفين فقط:
- `can_edit_prices_in_invoice`: تعديل أسعار البيع يدوياً داخل الفاتورة.
- `can_sell_below_cost`: البيع بأقل من سعر التكلفة المحددة.
- `show_invoice_profitability`: إظهار ربحية الفاتورة للمستخدم (تُحجب عادةً عن الكاشير).
- `manage_delivery`: متابعة توصيل الطلبات.
- `manage_purchase_returns`: إجراء مرتجعات المشتريات للموردين.

### ج. صلاحيات المسوق (Affiliate Permissions)
تُدار في لوحة تحكم منفصلة `/affiliate/*`:
| الصلاحية | الوصف |
|---|---|
| ✅ رؤية رابطه وكوده الخاص | في لوحة تحكم منفصلة `/affiliate/dashboard` |
| ✅ رؤية قائمة عملائه فقط | بدون أي بيانات تفصيلية للمحلات |
| ✅ رؤية عمولاته وحالتها | Pending / Approved / Paid |
| ✅ طلب سحب الرصيد | بعد الوصول لـ 500 ج.م |
| ❌ الدخول لأي بيانات Tenant | محظور تماماً (عزل تام) |

### د. صلاحيات المدير العام (Super Admin)
تُدار في لوحة التحكم المركزية `/admin/*`:
| الصلاحية | الوصف |
|---|---|
| ✅ إدارة الـ Tenants | تفعيل/إيقاف المحلات المشتركة |
| ✅ إدارة الباقات والأسعار | تعديل خطط الاشتراك (Plans) |
| ✅ إدارة المسوقين وعمولاتهم | اعتماد/رفض طلبات السحب للمسوقين |
| ✅ رؤية تقارير إيرادات SaaS | إجمالي المشتركين، الإيرادات الكلية |
| ❌ رؤية بيانات داخل المحلات | محظور - لا يمكنه رؤية مبيعات أو عملاء المحل |

## 9. موديول التقارير والتحليل (Reports & Analytics)

### أ. تقرير تحليل المبيعات (Sales Analysis)
- **المحركات (Filters):** التجميع حسب (اليوم، الشهر، الصنف، العميل، المخزن، المندوب).
- **الأعمدة:** الفترة، عدد الفواتير، صافي الكمية، صافي الإجمالي، صافي التكلفة، صافي الربح.

### ب. حركة الصنف التفصيلية (Item Movement Ledger)
- **الهدف:** تتبع الـ Life Cycle الكاملة للصنف وتأثيرها على التكلفة.
- **الأعمدة:** التاريخ، نوع الحركة، كمية وارد، كمية صادر، الرصيد التراكمي (Running Balance)، سعر التكلفة، سعر البيع.

### ج. حركة الخزينة (Treasury Movement)
- **الميزة:** عرض "رصيد الخزينة الحالي" بشكل بارز (مؤشر بصري أخضر).
- **الأعمدة:** رقم الحركة، الوصف، المرجع، الوقت الدقيق، الوارد (+)، الصادر (-)، الرصيد التراكمي.
